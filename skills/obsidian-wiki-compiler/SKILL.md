# Obsidian 知识库编译器

> 灵感来源：Andrej Karpathy 的 LLM Knowledge Base 方法论
> 核心理念：原始资料经 LLM "编译"成 Markdown Wiki，自动维护索引、链接和概念条目，让知识库从文件收纳箱变成可查询、会生长的第二大脑。

<trigger>
以下场景触发此 skill：
- "帮我整理我的 Obsidian"
- "更新一下知识库"
- "给笔记加链接"
- "扫描一下我的笔记"
- "知识库健康度检查"
- "编译 Wiki"
- 任何涉及 Obsidian 笔记整理、关联、维护的请求
</trigger>

## 核心理念

这个 skill 将 Karpathy 的 LLM Knowledge Base 方法论落地到用户的 Obsidian 库：

```
原始笔记（日记、收藏、灵感）
       ↓ LLM 编译
Wiki 概念页（索引、摘要、反向链接）
       ↓ 持续增强
可查询的知识网络（越用越厚）
```

**三条铁律：**
1. **原始内容不污染** — 用户写的原文只在末尾追加链接，绝不改动正文
2. **AI 生成内容有标记** — Wiki 概念页在 frontmatter 标注 `source: ai-compiled`，和用户原创内容区分
3. **链接必须真实** — 每个 `[[wikilink]]` 指向的文件必须真实存在，绝不创建指向不存在文件的链接

## 路径约定

| 路径 | 说明 |
|------|------|
| `~/Library/Mobile Documents/iCloud~md~obsidian/Documents/` | Obsidian 库根目录（以下简称 `$VAULT`） |
| `$VAULT/资源库/Wiki/` | Wiki 概念页存放目录（AI 编译产物） |
| `$VAULT/日记/` | 日记目录 |
| `$VAULT/灵感库/` | 个人思考和创作草稿 |
| `$VAULT/收藏夹/` | 外部内容剪藏（推特、文章） |
| `$VAULT/资源库/` | 工具、教程、提示词、自媒体、开发等 |
| `$VAULT/INDEX.md` | 全库内容索引 |

## 执行流程

用户触发后，按以下 Phase 顺序执行。每个 Phase 完成后向用户简要汇报进度。

### Phase 0: 现状扫描

**目标：** 了解知识库当前状态，为后续操作提供数据基础。

1. 扫描 `$VAULT` 下所有 `.md` 文件（排除 `.obsidian/` 目录）
2. 统计每个文件夹的笔记数量
3. 统计已有 `[[wikilink]]` 的笔记数量 vs 没有任何链接的笔记数量
4. 读取 `$VAULT/资源库/Wiki/` 下已有的概念页（如果有）
5. 记录上次编译时间（检查概念页 frontmatter 中的 `updated` 字段）

**产出：** 一份简要的库状态报告，告诉用户当前有多少笔记、多少孤岛、多少概念页。

**并行策略：** 使用 3 个 Explore 子代理分别扫描：
- 代理 1：灵感库 + 收藏夹（读取所有 .md 内容，提取主题关键词和已有 wikilink）
- 代理 2：资源库所有子目录（同上）
- 代理 3：日记目录（读取所有日记，提炼有价值的洞察）

### Phase 1: 概念发现

**目标：** 从所有笔记中识别出高频概念，决定要创建或更新哪些 Wiki 概念页。

1. 基于 Phase 0 的扫描结果，找出在 **3 个以上不同文件** 中出现的概念/工具名/人名/方法论
2. 与已有的 Wiki 概念页对比：
   - 新概念 → 标记为"待创建"
   - 已有概念但有新关联笔记 → 标记为"待更新"
   - 已有概念且无变化 → 跳过
3. 将发现的概念列表呈现给用户确认，再进入下一步

**概念筛选标准：**
- 必须是具体的、可链接的概念（如"Claude Code"、"视频制作"），不能是过于笼统的词（如"学习"、"工作"）
- 优先选择能串联多个领域笔记的"桥梁概念"

### Phase 2: Wiki 编译

**目标：** 创建或更新 Wiki 概念页。

**概念页模板：**

```markdown
---
type: 概念
source: ai-compiled
aliases: [别名1, 别名2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
---

# 概念名

> 一句话定义（简洁直白，非程序员也能看懂）

## 相关笔记

- [[笔记名]] — 一句话说明这篇笔记和本概念的关系
- [[笔记名]] — 一句话说明

## 日记洞察

- **YYYY-MM-DD**：从日记中提炼的与本概念相关的认知或经验（摘录关键原文）

## 关键要点

- 从所有相关笔记中提炼的核心认知（不是复述，是升华）
- 每条要点一句话，直击本质
```

**规则：**
- 新建概念页：按模板创建，填充所有已知关联
- 更新概念页：只追加新发现的笔记链接和日记洞察，不删除已有内容
- `aliases` 要包含常见的变体写法（中英文、缩写）
- `相关笔记` 中的每个 `[[链接]]` 必须指向真实存在的文件
- `日记洞察` 只提炼有认知价值的内容（原创观点、方法论、经验教训），忽略流水账

### Phase 3: 反向链接注入

**目标：** 给缺少 wikilink 的笔记添加反向链接，让孤岛笔记接入知识网络。

**操作方式：**
1. 在目标笔记的 **末尾** 追加 `## 相关笔记` 区块
2. 如果笔记末尾已有 `## 相关笔记`，则在已有内容下追加新链接，不重复
3. 绝不修改笔记的原始正文内容

**链接格式：**
```markdown
## 相关笔记

- [[目标笔记名]] — 一句话说明关联原因
```

**优先级：**
1. 收藏夹中完全没有链接的笔记（最高优先）
2. 灵感库中链接不完整的笔记
3. 资源库中缺少跨目录链接的笔记

**并行策略：** 使用多个子代理分别处理不同目录，每个代理负责一个文件夹的链接注入。

### Phase 4: 日记提炼

**目标：** 从日记中提取有价值的洞察，链接到对应的 Wiki 概念页。

**什么值得提炼：**
- 原创认知和观点（对现象的独特理解）
- 方法论和经验总结（"我发现 X 的最好方式是 Y"）
- 项目决策和复盘（为什么选了 A 而不是 B）
- 有启发性的失败和反思

**什么不提炼：**
- 纯生活流水账（吃了什么、去了哪里）
- 情绪宣泄（除非包含有价值的自我认知）
- 已经在其他笔记中完整记录的内容（避免重复）

**提炼格式：**
每条洞察附带日期和原文关键句，写入对应概念页的"日记洞察"区块。

### Phase 5: 健康度检查

**目标：** 发现知识库中的问题并报告给用户。

检查项目：

| 检查项 | 说明 |
|--------|------|
| 孤岛笔记 | 没有任何 `[[wikilink]]` 指向它，也不包含任何 `[[wikilink]]` 的笔记 |
| 断裂链接 | `[[链接]]` 指向不存在的文件 |
| 重复内容 | 两篇笔记内容高度相似（如收藏夹和灵感库中的同名文件） |
| 概念页过时 | 有新笔记提到某概念但概念页未更新 |
| 空文件 | 文件存在但内容为空或只有标题 |

**产出：** 一份健康度报告，按严重程度排序，建议用户如何处理每个问题。不自动修复——报告给用户决定。

## 输出回归

这是 Karpathy 方法论的核心环节：**每次使用知识库产生的有价值输出，都应该回归到知识库中。**

当用户通过 Claude Code 做研究、分析、对比后，主动提醒用户：

> "这次分析的结论要存回 Obsidian 吗？我可以帮你归档到对应的概念页或创建新笔记。"

回归的内容标记 `source: ai-generated`，和用户原创内容区分。

---

## 权限边界

### 可以做
- 在笔记末尾追加 `## 相关笔记` 区块
- 在 `资源库/Wiki/` 下创建和更新概念页
- 读取所有 `.md` 文件内容用于分析
- 更新 `INDEX.md`

### 绝不可以做
- 修改任何笔记的原始正文内容
- 删除任何文件或文件夹
- 在 `资源库/Wiki/` 以外的地方创建新文件
- 修改 `.obsidian/` 目录下的任何配置
- 添加指向不存在文件的 `[[wikilink]]`

---

## 执行策略

- **首次运行**：全量扫描 + 全量编译（Phase 0-5 完整执行）
- **后续运行**：增量更新（检查概念页的 `updated` 字段，只处理有变化的部分）
- **并行处理**：Phase 0 的扫描和 Phase 3 的链接注入使用多个子代理并行，加快速度
- **用户确认点**：Phase 1 结束后暂停，让用户确认概念列表再继续；Phase 5 的健康度报告不自动修复

## 模型选择

此 skill 涉及大量文件读取和内容理解，建议使用 Opus 模型以获得最佳的语义理解和关联发现能力。子代理扫描任务可以使用 Sonnet 以节省成本。
